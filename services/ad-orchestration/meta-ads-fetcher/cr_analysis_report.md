# 広告CR分析レポート: embedding × 配信データの検証

## 分析概要

歯科矯正マウスピース広告（ローコスト）のCRについて、Gemini Embedding 2（3072次元）と配信実績データを組み合わせてCPA予測・CR制作示唆の抽出を試みた。

| データ | 規模 |
|--------|------|
| 対象プロジェクト | ローコスト（歯科矯正マウスピース） |
| embedding完了CR | 281本 / 396本 |
| 配信実績あり | 163本 |
| CV有りCR | 33本 |
| 使用embedding | Gemini Embedding 2 (3072d) — 動画全体/テキスト(テロップ+書き起こし)/シーン単位の3種類 |
| 構成ラベリング | Gemini Flash 2.5 Lite — 10軸×195本をJSON構造化出力で分類 |

---

## 1. embeddingの情報構造（PCA分解）

281本のCRの3072次元embeddingをPCAで分解した結果、以下の10主成分が全体の61%を説明。

| PC | 寄与率 | 軸名 | +側 | -側 |
|----|--------|------|-----|-----|
| PC1 | 28.9% | 訴求コンセプト | Eライン・横顔を直接訴求（問いかけ型） | 垢抜け・雰囲気変化を訴求（友達の変化型） |
| PC2 | 7.6% | 撮影トーン | 束河・奥山系の美容寄りトーン | 松永系の実用寄りトーン |
| PC3 | 5.6% | 視点の距離 | 本人の悩み（姉妹会話） | 他人を観察（彼氏目線） |
| PC4 | 4.4% | 映像表現 | アニメ会話劇 | 実写UGC |
| PC5 | 3.5% | 感情vs理性 | 恋愛・モテ（感情訴求） | コスパ・価格（理性訴求） |
| PC6 | 3.2% | ターゲット性別 | 女性同士の会話 | 男性視点 |
| PC7 | 2.5% | 信頼の根拠 | 比較検討・スペック訴求 | 推薦・共感訴求 |
| PC8 | 2.4% | 展開速度 | BA直球・短尺・すぐ結論 | ストーリードラマ・長尺 |
| PC9 | 2.0% | 時間軸 | 矯正前の期待（これから） | 矯正後の振り返り（やった結果） |
| PC10 | 1.8% | 商材ポジション | ファスト矯正（手軽さ） | 美容ジェネリック（洗練） |

**類似度の29%は「訴求コンセプト」1軸で決まる。** 映像表現（アニメ/実写）は4.4%しかなく、embeddingは映像スタイルよりも「何を言っているか」を重視している。

---

## 2. embeddingが捉えられないもの

### 構成順序は完全に潰れている

「悩み→提案→権威→オファー」（27本）と「悩み→提案→実績→オファー」（14本）がembedding上でほぼ同じ位置にいる。権威性を前半に置くか後半に置くかで分割しても、PC1〜10のどれも反応しない（効果量ほぼゼロ）。

Gemini Embeddingはbag-of-conceptsとして動画を捉えており、要素の出現順序を保持しない。

### ただし「展開速度」は擬似的に構成情報を含む

PC8（展開速度）は「結論までの距離」を捉えており、これは順序そのものではないが構成の一側面。尺・シーン密度・テロップの出方の複合として、embeddingに含まれている。

## 3. アニメ vs 実写の分離

### そのままの類似度

| 比較 | 平均類似度 |
|------|-----------|
| アニメ同士 | 0.925 |
| Vlog同士 | 0.954 |
| アニメ↔Vlog | 0.883 |
| 分離度 | 0.057（小さい） |

### 重心除去後（「歯科矯正広告」の意味を引いた残差）

| 比較 | 平均類似度 |
|------|-----------|
| アニメ同士 | 0.460 |
| Vlog同士 | 0.188 |
| アニメ↔Vlog | -0.245 |
| **分離度** | **0.569（10倍に向上）** |

embeddingに表現差分の情報がないのではなく、意味成分が支配的すぎて埋もれていた。重心除去で表現軸が明確に分離される。

## 4. embedding × CPA予測の検証

### CV有りCRのみ（n=31）

| 指標 | 値 |
|------|-----|
| R² | 47.5% |
| 調整R² | 21.2% |

一見効いているように見えるが、31本で10変数は過学習。交絡チェックで以下が判明：

- PC7「共感型がCPA良い」→ 他軸の偏りで説明がつき、独立効果は¥276/σまで縮小
- PC8「直球型がCPA良い」→ 交絡除去後も¥861/σ残るが、CV加重で消える
- PC1「垢抜け文脈がCPA良い」→ 独立効果¥3,920/σで最も堅い

### 全配信CR（n=163、CV=0含む）

| 指標 | 値 |
|------|-----|
| R² | 4.8% |
| **調整R²** | **-1.5%** |

**nを増やしたら説明力がゼロになった。** CV有り31本で見えた傾向はノイズだった。

### CV有無の判別（効果量d）

| 軸 | 効果量d |
|----|--------|
| PC6: ターゲット性別 | +0.28 |
| PC9: 時間軸 | -0.26 |
| PC3: 視点の距離 | +0.21 |
| PC8: 展開速度 | +0.19 |

全部 |d| < 0.3 で統計的に弱い。

## 5. シーン列比較の検証

動画全体embeddingではなく、5区間（冒頭/序盤/中盤/終盤/CTA）に正規化したシーンembedding列で比較。

### 結果

- CPA ¥23Kと¥158KのCRがシーン列類似度0.983
- 全区間でTop5基準との類似度とCPAに弱い正の相関（r=+0.16〜+0.27）
- つまり「似ているほどCPAが悪い」方向

**シーンembedding自体がWhatに偏っているため、列にしても同じ問題が発生する。**

## 7. embeddingの情報構造とCPA予測の相性

### なぜembeddingでCPA予測が効かないか

ローコストのCRは全部「歯科矯正が安くて品質良い」と言っている。What（何を言っているか）はほぼ同じ。CPAを決めるのはHow（どう伝えるか）とSequence（どの順番で伝えるか）。

| 差分の種類 | embeddingに含まれるか | CPA影響 |
|-----------|---------------------|---------|
| 言っている内容（What） | ○（PC1, 29%） | 弱い（同一商材で共通） |
| 映像フォーマット | ○（PC4, 4%） | あり（アニメ→CV=0） |
| 展開テンポ | △（PC8, 2%） | 弱い（加重で消える） |
| 冒頭3秒の掴み方 | △（PC1に混在） | あり（フック種別で¥13K差） |
| **要素の順序** | **✗** | **不明（データ不足）** |
| **テロップの言い回し** | **✗** | **おそらくあり** |
| **声のトーン・間** | **✗** | **おそらくあり** |
| **フォント・色・配置** | **✗** | **おそらくあり** |

### embeddingがCPA予測に効く条件（仮説）

1. **「何を言っているか」自体が多様な場合** — 異業種CR混在データセットなら効く可能性
2. **大量の教師データ（n=1000+ CV有り）** — 微弱シグナルを機械学習で拾える可能性
3. **タスク特化fine-tune** — CPA予測用にembeddingを学習し直す（ただしデータ不足）

### embedding vs 構成ラベルの役割分担

| | embedding | 構成ラベル |
|--|-----------|-----------|
| 何を言っているか | ○ | ○ |
| 映像表現 | ○ | △ |
| 展開テンポ（擬似構成） | △ | △ |
| **伝える順番** | **✗** | **○** |
| **CPA予測力** | **R²≒0%** | **フック種別等で¥13K差検出** |
| CR検索・類似発見 | ◎ | ✗ |
| 重複検出 | ◎ | ✗ |

---

## 8. 実用的な結論

### embeddingの使い道

- **CR検索**: 「この動画に似たCR」を即座に発見
- **表現スタイル分類**: 重心除去でアニメ/実写/UGC等を分離
- **重複検出**: リメイク・素材変更の類似度0.98+で検出
- **CPA予測には使えない**

### 構成ラベルの使い道

- **制作指針**: 「問いかけフック × 無料カウセ × 会話劇 × 実写」が現時点の最適構成
- **CPA差分分析**: フック種別、オファー種別、ストーリー構造でCPA差を検出
- **ただしn=33のCV有りCRでの傾向であり、断定には慎重を要する**

### 次のステップ

1. embedding全本完了を待って再検証（281/396 → 396/396）
2. CV有りCRが増えるのを待つ（33→50+で重回帰の信頼性向上）
3. 構成ラベルをDBに正式格納し、入稿時の制作指針として運用
4. 構成ラベル × CPA のクロス分析（フック×オファーの組み合わせ等）はnが増えてから
