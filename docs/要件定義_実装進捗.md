## イシュー一括作成

PAT に `workflow` スコープがない環境では `.github/workflows` のプッシュが拒否されるため、**ローカルから GitHub API で作成**する。

### 原因（よくあるエラー）

`export GITHUB_TOKEN='（Fine-grained なら…）'` のように **説明文や日本語を代入**すると、HTTP ヘッダーに使えず次のエラーになる。

`TypeError: Cannot convert argument to a ByteString...`

**対策:** トークンには GitHub が発行した **半角の `ghp_...` / `github_pat_...` のみ**を入れる。説明文は **export 行に書かない**。

---

### 手順 A（推奨）GitHub CLI — トークンをターミナルに貼り付けない

次を **そのまま順に** コピペ実行する。

```bash
brew install gh
gh auth login
```

対話では GitHub.com / HTTPS / ブラウザでログインを選ぶ。

```bash
cd /Users/bonnou_p005/projects/ureru-buzz-ai
unset GITHUB_TOKEN
node scripts/create-requirement-gap-issues.mjs
```

`pnpm issues:requirement-gap` でも同じ。

---

### 手順 B（代替）Classic または Fine-grained PAT

1. ブラウザで GitHub → **Settings → Developer settings → Personal access tokens** からトークンを作成する。  
   - Fine-grained の場合: 対象リポジトリを選び、**Issues: Read and write**。
2. 表示された **トークン文字列だけ**（`ghp_` または `github_pat_` で始まる）をコピーする。

```bash
cd /Users/bonnou_p005/projects/ureru-buzz-ai
unset GITHUB_TOKEN
export GITHUB_TOKEN=ペーストしたトークンだけをここに半角で貼る
node scripts/create-requirement-gap-issues.mjs
```

`export` の行には **引用符で囲んだ説明文を入れない**（全角括弧も不可）。

---

### Git の origin から PAT を外す（推奨）

リモート URL に `https://USER:TOKEN@github.com/...` が入っていると漏えいリスクがある。HTTPS のきれいな URL に戻し、認証は **gh** に任せる。

```bash
cd /Users/bonnou_p005/projects/ureru-buzz-ai
git remote -v
git remote set-url origin https://github.com/ZENNOU-dev/ureru-buzz-ai.git
gh auth setup-git
```

以前 URL に含めた PAT は **GitHub 上で失効（Revoke）** し、必要なら新しいトークンを `gh auth login` で使う。

---

## 補足: 認証・シークレット（エンジニア向け）

### なぜ「日本語を入れたら」fetch が落ちるか

- ブラウザ／Node の `fetch()` は、**リクエストヘッダーの値を HTTP 仕様に沿ったバイト列**として扱う経路がある（WHATWG では一部ヘッダー値を *ByteString* と定義。実質 **Latin-1（0–255）** の範囲）。
- `Authorization: Bearer …` に **Unicode（日本語・全角括弧など）** を載せると、内部で「255 を超えるコードポイントは ByteString にできない」となり、`TypeError: Cannot convert argument to a ByteString` になる。
- **GitHub の PAT は ASCII のみ**なので問題ない。失敗例は「説明文をトークンだと思って代入した」ケースがほぼすべて。

### ローカルで何を使うか（対比）

| 方式 | 概要 | 向いている用途 |
|------|------|----------------|
| **`gh auth login`** | OAuth 系。トークンは OS のキーチェーン等に保存され、`gh auth token` が短命トークンを返す。シェル履歴に長期 PAT を残しにくい。 | **個人マシンからの手動スクリプト（推奨）** |
| **`export GITHUB_TOKEN=PAT`** | プロセス環境変数。シェル履歴・`ps`・子プロセスに露出しうる。値は **ASCII の PAT のみ**。 | CI の注入、一時的な自動化 |
| **リモート URL 埋め込み** | `https://x-access-token:TOKEN@github.com/...` など。**避ける**。ログ・設定共有で漏えいしやすい。 | 非推奨 |

### 「Google でトークン管理」＝ Google Cloud Secret Manager なら進められるか

**結論: 進められる。ただし主戦場は「GCP 上で動く処理」か「取得してから環境変数に載せる運用」**。

- **Secret Manager**（GCP）: シークレットの保管・バージョン管理・監査ログ。読み取りは **IAM + 実行主体**（Cloud Run / Cloud Functions のサービスアカウント、または `gcloud auth` 済みの開発者）が行う。
- **このリポジトリの Issue 作成スクリプトをローカルだけで回す**場合の典型パターン:
  1. PAT を Secret Manager に登録（例: シークレット ID `github-pat-issues`）。
  2. 開発者が `gcloud` で認証済みかつ `secretAccessor` 権限を持っているとき、**実行直前にだけ**値を取り出して環境変数に載せる。

```bash
# 例: アクセサ権限と gcloud 認証が済んでいる場合のみ
export GITHUB_TOKEN="$(gcloud secrets versions access latest --secret=github-pat-issues)"
node scripts/create-requirement-gap-issues.mjs
```

- **注意**: 上記でも実行中のプロセスには平文が載る。履歴に残したくないなら **Space 始まり（bash で履歴除外）**や、**対話入力**、**1Password CLI / Doppler** などと同列の「取り出し直し」運用として扱う。
- **本番らしい形**: Issue 作成を **Cloud Run / Cloud Functions / Cloud Build** に載せ、Secret Manager を **環境変数としてマウント**（またはランタイム API で取得）し、GitHub API をサーバーから呼ぶ。ローカルに PAT を置かない。

### GitHub 側だけで閉じる場合

- **GitHub Actions** の **Repository secrets** に PAT または `GITHUB_TOKEN`（権限はジョブの `permissions` で制御）を置き、ワークフロー内で `github.rest.issues.create` する方法もある（`.github/workflows` のプッシュには `workflow` スコープ付き認証が必要な点は別途注意）。

### 推奨運用: Secret Manager ＋ Cloud Run Job（端末に PAT を常駐させない）

PAT は **Secret Manager にのみ**保存し、**Cloud Run Job** 実行時に `GITHUB_TOKEN` として注入して同じ Node スクリプトを走らせる。手順は次を参照。

- **`deploy/github-issue-job/README.md`** — シークレット作成、Artifact Registry、Job デプロイ、`gcloud run jobs execute`
- **`deploy/github-issue-job/Dockerfile`** / **`cloudbuild.yaml`** — イメージビルド

コンテナ内では **`gh` は使わず**、`CLOUD_RUN_JOB` / `K_SERVICE` 検出時は Secret 注入の `GITHUB_TOKEN` のみ有効。
